# План разработки: iVision Conf TMA

**Версия:** 2.0
**Дата:** Март 2026
**Стек:** Python (FastAPI) + aiogram 3 · React + Vite + @telegram-apps/sdk · Supabase (PostgreSQL) · Vercel + Railway

---

## Что такое MVP

**Цель MVP:** запустить ближайшую конференцию на собственном боте. Участник приходит → регистрируется → видит программу → участвует в реферальной игре.

**MVP включает:**
- Бот открывает GetCourse-лендинг через TMA-кнопку
- GetCourse webhook → создаёт участника в базе
- TMA: 4 вкладки (видны все), [Программа] и [Игра] работают
- Lock Screen для [Розыгрыш] и [Игра] → проверка подписок → разблокировка
- [Игра]: Уровень 0 (анкета) + реферальная ссылка + прогресс-шкала призов
- [Программа]: расписание + профили спикеров (без LIVE-баннера и без парсера в MVP)
- Базовые уведомления бота: приветствие + реф. событие

**MVP НЕ включает (добавим после):**
- [Розыгрыш] ← уже есть в SaleBot, переносим отдельной задачей
- [Услуги] ← добавляем в самом конце
- Веб-админка (MVP = заносим данные напрямую в Supabase)
- LIVE-баннер и таймер обратного отсчёта
- Рассылки-знакомства со спикерами
- Ветеранский путь
- Состояние 3 (после конференции)

---

## Архитектура MVP-системы

```
Telegram-бот (@medialift_app_bot)
  │
  ├── /start → кнопка «Открыть iVision» → открывает TMA
  │
  └── Уведомления участникам (приветствие, рефералы, призы)

TMA (React + Vite) — хостинг: Vercel
  │
  ├── Незарегистрирован → кнопка «Зарегистрироваться» → GetCourse лендинг в браузере
  │
  └── Зарегистрирован → 4 вкладки
        [Программа] [Услуги*] [Розыгрыш] [Игра]

Backend API (FastAPI) — хостинг: Railway
  │
  ├── POST /webhook/getcourse      ← webhook от GetCourse при регистрации
  ├── POST /webhook/getcourse/form ← webhook от GetCourse при заполнении анкеты
  ├── GET  /api/conference/current ← данные конференции
  ├── GET  /api/sessions           ← расписание
  ├── GET  /api/speakers           ← спикеры
  ├── GET  /api/participant/me     ← профиль участника
  ├── GET  /api/referral/stats     ← реф. статистика
  └── POST /api/subscriptions/verify ← проверка подписок

База данных: Supabase (PostgreSQL)
  Схема: referalka-таблицы + iVision-расширение (brief.md, раздел 9)
```

## Фаза 1: База данных

**Итог:** полная схема БД создана в Supabase. Готова к работе.

### Структура таблиц

Основа — схема referalka (`telegram_users`, `event_participants`, `referral_links`, `referral_events`, `gifts`, `gift_issuances`, `notifications_log`) + iVision-расширения:

**Новые поля в `telegram_users`:**
- `has_filled_form` BOOLEAN — прошёл Level 0

**Новые поля в `event_participants`:**
- `channels_verified` BOOLEAN — прошёл «Клубный взнос»

- `first_name` VARCHAR — из GetCourse
- `last_name` VARCHAR — из GetCourse
- `phone` VARCHAR — из GetCourse
- `getcourse_id` VARCHAR — ID в GetCourse

**Новые таблицы:**
`conferences` · `speakers` · `sessions` · `broadcast_messages` · `commercial_items` · `secret_codes` · `speaker_applications`

(Полная схема полей — в [brief.md, раздел 9](brief.md))

## Фаза 2: GetCourse Webhook + Авторизация

**Итог:** человек регистрируется на GetCourse → TMA автоматически разблокирует кабинет.

- [ ] Создаёт/обновляет запись в `telegram_users` + `event_participants`
- [ ] Привязывает `ref_code` → начисляет тикет рефереру
- [ ] Бот присылает приветственное сообщение с кнопкой «Открыть приложение»

### 2.2 Авторизация в TMA

- [ ] TMA: при старте вызывает авторизацию, сохраняет JWT в памяти
- [ ] Роутинг: `unregistered` → экран лендинга; `registered` → 4 вкладки

**Итог:** [Розыгрыш] и [Игра] заблокированы до подписок. После проверки — оба разблокируются + 1 билет.

- [ ] Компонент `LockScreen.tsx`:
  - Текст: «Подпишитесь на каналы, чтобы открыть розыгрыш и реферальную игру»
  - Список каналов с кнопками «Подписаться» (открывают канал через `openTelegramLink()`)
  - Кнопка «Проверить подписки и забрать подарки»
- [ ] Бэкенд `POST /api/subscriptions/verify`:

- [ ] TMA: после успеха → pop-up «+1 билет! Розыгрыш открыт» → редирект на [Игра]
- [ ] Если вкладка [Розыгрыш] или [Игра] → `channels_verified = false` → показать `LockScreen`
