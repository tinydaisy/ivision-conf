# Brief: iVision Conf — Telegram Mini App для онлайн-конференции

**Версия:** 1.0
**Дата:** Март 2026
**Статус:** Ready for Development

---

## 1. Суть продукта

Telegram Mini App для участников онлайн-конференции iVision. Открывается по ссылке прямо в Telegram — без скачивания приложений, без отдельной регистрации. Авторизация через Telegram-аккаунт автоматически.

**Уникальная позиция:** единственное решение на рынке СНГ, которое объединяет конференц-функционал (программа, спикеры) + реферальную геймификацию + Telegram-нативность в одном продукте.

---

## 2. Целевая аудитория

**Участник:** предприниматель, эксперт, специалист в нише iVision. Аудитория СНГ. Основной канал — Telegram.

**Организатор (внутренний пользователь):** команда iVision — управляет контентом через веб-админку.

---

## 3. Бизнес-контекст

- Конференция проводится периодически (минимум 3-4 раза в год)
- Каждая конференция — 2 дня, один последовательный поток докладов
- Регистрация участников — через GetCourse или сторонние лендинги (существующая инфраструктура)
- TMA интегрируется с GetCourse через Webhook: факт регистрации на GetCourse открывает полный кабинет в TMA
- Бот конференции — отдельный от сейл-бота, собирает собственную аудиторию как актив организатора
- После конференции база участников остаётся доступной для последующих рассылок

---

## 4. Архитектура состояний

### Состояние 1: Незарегистрирован
Пользователь переходит по ссылке → сразу попадает на сторонний лендинг конференции:
- после регистрации там его возвращает в бот и ему в приложении открываются дополнительные разделы меню 

### Состояние 2: Зарегистрирован (до и во время конференции)
Bottom Tab Bar: **[Программа] [Спикеры] [Розыгрыш] [Игра]**
- Программа: расписание по дням, карточки докладов, кнопки "Получить записи", "Заявка на спикерство"
- Спикеры: список с фото, bio, названием подарка после доклада, названием подарка для розыгрыша, ссылкой на телеграм-канал. Ссылки на подарки хранятся в базе данных, но в этих разделах не выводятся, кнопки "Получить записи", "Заявка на спикерство"
- Призы: информация о разыгрываемых подарках, выдача 1 билета на розыгрыш за подписку на каналы всех спикеров, возможность ввести кодовое слово и получить дополнительный билет на розыгрыш
- Игра: реферальная механика - информация о доступных подарках за количество приведенных людей, реферальная ссылка, проверка перешедших и зарегистрированных по ссылке, макеты и тексты для анонсов, инструкции по анонсированию


### Состояние 3: После конференции
Bottom Tab Bar: **Календарь событий площадки iViSiON**
- В заданную дату завершения конференции все разделы [Программа] [Спикеры] [Розыгрыш] скрываются и открывается раздел [Календарь]

## 5. Функциональные требования

### 5.1 Лендинг (незарегистрированный)
- Отображение по ссылке стороннего лендинга внутри мини-аппа

### 5.2 Программа конференции
- Переключение между днями (День 1 / День 2)
- Карточка доклада: время, название, спикер (маленькое фото + имя и фамилия), трек (цветовая метка, опционально), карточка кликабельна и ведет на карточку спикера
- Обновление в реальном времени (программа может меняться до последнего)

### 5.3 Спикеры
- Список карточек: фото, имя и фамилия, профессия, тема доклада
- Клик → детальная страница спикера: фото, ссылки на соцсети, регалии списком, тема доклада, название подарка после эфира, название подарка для розыгрыша,  
- Раздел полностью скрывается после конференции (управляется флагом в БД о завершении конференции)

### 5.4 Розыгрыш
- дата розыгрыша
- список разыгрываемых подарков
- кнопка "получить билет"
- кнопка "смотреть билеты"
- клик на "получить билет" : если не подписан на всех спикеров - то заставиьт подписаться и выдать первый билет. Если уже подписан - то отобразить его список билетов и кнопку "Получить еще билет"
- клик на "Получить еще билет": информация что человек должен ввести кодовое слово, которое получил на эфире + кнопка "Ввести кодовое слово"
- клик на "Ввести кодовое слово": пользователь вводит кодовое слово - оно проверяется с перечнем доступных слова (регистр не зависит) и теми словами которые пользователь уже вводил. Если такое слово доступно и он его еще не вводил - выдается билет и запоминается в бд. Если вводил уже или нет такого слова - то выводится ему предупреждение и предлагается ввести снова

### 5.5 Игра
- берется движок из проекта referalka


### 5.6 Уведомления (через бота)
- При регистрации реферала: "Иван зарегистрировался по твоей ссылке! +1 тикет"
- Напоминания: за 3 дня, 1 день, 30 минут до конференции
- При достижении порога призов
- Рассылки после конференции (материалы, итоги розыгрыша)

## 6. Веб-админка организатора

Закрытая паролем страница `[домен]/admin`. Один пользователь (команда iVision).

### 6.1 Раздел "Конференция"
- Название, дата, описание, статус (draft / active / finished)
- Флаг "скрыть спикеров" (включается после конференции)

### 6.2 Раздел "Программа"
**Ввод через шаблон с разделителями:**
```
10:00 | Открытие конференции | Марго Форбс | | Добро пожаловать!
10:30 | Маркетинг в 2026 году | Иван Петров | Чек-лист по контенту
11:15 | Перерыв
11:30 | Продажи через контент | Анна Сидорова | Шаблон скриптов продаж
```
Формат колонок: `ЧЧ:ММ | Тема | Имя спикера | Подарок спикера | [Трек]`

- Вставить текстом → система парсит → показывает превью → кнопка "Сохранить"
- При повторном сохранении — **существующие записи обновляются** (не дубли). Спикер ищется по имени — если уже есть в базе, привязывается автоматически
- При сохранении: **авто-генерация рассылок** для всех новых/изменённых сессий
- Если сессия изменилась и по ней были вручную отредактированные рассылки → показывает предупреждение: "3 рассылки затронуты. Просмотреть?"
- Поддержка нескольких дней (вкладки "День 1" / "День 2")

### 6.3 Раздел "Спикеры"
**Профильные данные** (редко меняются, применяются везде автоматически):
- Список спикеров: фото-миниатюра, имя, должность
- Форма: фото (загрузка), афиша (загрузка), имя, должность, компания, bio, ссылки (TG-канал, Instagram, сайт)
- Кнопка "Копировать ссылку страницы спикера" → `[домен]/s/[slug]` — отправить спикеру для проверки
- Страница спикера публичная (без логина): авто-собирается из БД — фото, bio, расписание его выступлений с темами и временем

**Привязка сессий:**
- Под каждым спикером — список его сессий в текущей конференции (из `sessions`)
- Клик на сессию → переход к её редактированию

### 6.4 Раздел "Рассылки"
> Главная экономия времени. Все тексты генерируются из программы. Организатор только проверяет и, при необходимости, правит.

**Список всех запланированных рассылок** (авто-сгенерированных + ручных):
- Фильтр по типу: предстоящие / отправленные / требуют проверки
- Каждая строка: тип, дата отправки, первые 80 символов текста, статус, иконка "изменено вручную"
- Значок ⚠️ если `needs_review = true` (сессия изменилась после ручного редактирования)

**Редактирование рассылки:**
- Поле даты/времени отправки (редактируемо)
- Поле текста (редактируемо, textarea)
- Кнопка "Сбросить к авто-шаблону" (для вручную отредактированных)
- Кнопка "Отправить сейчас" (для срочных)

**Авто-шаблоны рассылок (примеры):**

`pre_5min` (за 5 минут до сессии):
```
⏰ Через 5 минут начинается выступление {speaker.name}!
Тема: {session.title}
Спикер подарит: {session.gift_description}
Подключайтесь к трансляции: {track.stream_url}
```

`post_thanks` (сразу после сессии):
```
👏 Спасибо {speaker.name} за выступление!
Тема была: {session.title}
Следите за программой конференции в приложении.
```

`day_start` (утро каждого дня):
```
🌅 Доброе утро! Сегодня День {day} конференции iVision.
Программа дня:
{session_list_for_day}
Не забудьте открыть приложение — там полное расписание.
```

**Каскадное обновление:** при изменении сессии (время, тема, подарок, трек) → система автоматически пересчитывает все не-отредактированные рассылки этой сессии. Отредактированные помечает флагом `needs_review`.

### 6.5 Раздел "Реферальная программа"
- Настройка призов: название, порог тикетов, описание
- Настройка начисления тикетов (за регистрацию, за реферала, за активность)

### 6.6 Раздел "Участники"
- Таблица: ник, имя, дата регистрации, тикеты, кол-во рефералов
- Выгрузка в CSV/Excel

### 6.7 Быстрые действия (через Telegram-бота организатора)
- `/stats` — текущая статистика участников и отправленных рассылок
- Уведомление при новой заявке спикера + кнопки "Одобрить" / "Отклонить"
- Уведомление если рассылка требует проверки (`needs_review`)

## 7. Интеграция с GetCourse

```
Участник нажимает "Зарегистрироваться" в TMA
→ Открывается GetCourse landing в браузере
→ Участник заполняет форму на GetCourse
→ GetCourse отправляет Webhook на наш backend:
  POST /webhook/getcourse
  { telegram_id: "12345", email: "user@mail.com", ref_code: "ABC" }
→ Backend меняет статус participant: interested → registered
→ TMA обновляется — открывается полный кабинет
→ Бот присылает приветственное сообщение
```

**Реферальная цепочка через GetCourse:**
- При переходе по реф. ссылке `startapp=REF_CODE` → бот запоминает ref_code
- При Webhook от GetCourse → ref_code привязывается к участнику
- Реферер получает +1 к счётчику + тикет

## 8. Техническая архитектура

### Компоненты
| Компонент | Назначение |
|---|---|
| `@ivision_conf_bot` | Telegram-бот, точка входа, уведомления |
| TMA (фронтенд) | React/Vue SPA, Telegram Web Apps API |
| Веб-админка | Простой SPA или SSR на том же бэкенде |
| Бэкенд API | Node.js / Python, REST API |
| БД | PostgreSQL (схема совместима с referalka) |
| Хранилище медиа | S3 / Cloudflare R2 для фото спикеров |

### Telegram Web Apps API
- `initData` — авторизация без формы
- `startapp` параметр — реферальный код
- `MainButton` — главный CTA
- `BackButton` — нативная кнопка назад
- `shareURL` — шеринг реф. ссылки
- `HapticFeedback` — вибрация при ключевых действиях
- `expand()` — полноэкранный режим
- CSS переменные `--tg-theme-*` — адаптация под тему пользователя

## 9. Схема БД (совместима с referalka)

### Таблицы из referalka (переиспользуются без изменений)
- `telegram_users` — глобальный реестр пользователей
- `event_participants` — факт участия в событии
- `referral_links` — реф. ссылки
- `referral_events` — лог переходов
- `gifts` — призы/пороги
- `gift_issuances` — выдачи призов
- `notifications_log` — лог уведомлений

### Новые таблицы (конференц-блок)

**`conferences`** — данные конференции (1 активная запись)
| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `event_id` | UUID → events | Связь с таблицей events |
| `title` | VARCHAR | Название конференции |
| `subtitle` | TEXT | Подзаголовок |
| `description` | TEXT | Описание для лендинга |
| `start_date` | DATE | Дата начала |
| `end_date` | DATE | Дата окончания |
| `timezone` | VARCHAR | Часовой пояс |
| `status` | ENUM | `draft` / `active` / `finished` |
| `speakers_hidden` | BOOLEAN | Скрыть раздел спикеров |
| `landing_hero_image_url` | TEXT | Обложка лендинга |

**`tracks`** — тематические треки (опционально, может быть 1 "Основная программа")
| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `conference_id` | UUID → conferences | К какой конференции |
| `title` | VARCHAR | Название трека |
| `color` | VARCHAR | HEX-цвет метки |
| `stream_url` | TEXT | Ссылка на трансляцию этого трека (используется в рассылках) |
| `sort_order` | INTEGER | Порядок отображения |

**`speakers`** — спикеры
> **Архитектурный принцип:** спикер может выступать несколько раз в одной конференции (разные темы / дни). Профильные данные хранятся в `speakers` один раз. Тема, подарок, время — в каждой сессии отдельно. Один раз правишь bio или фото — обновляется везде.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `conference_id` | UUID → conferences | К какой конференции |
| `slug` | VARCHAR | URL-идентификатор (UNIQUE): `ivisionconf.ru/s/ivan-petrov` |
| `name` | VARCHAR | Полное имя |
| `title` | VARCHAR | Должность/звание |
| `company` | VARCHAR | Компания |
| `bio` | TEXT | Биография |
| `photo_url` | TEXT | Фото (загружается в R2/S3) |
| `poster_url` | TEXT | Индивидуальная афиша спикера |
| `telegram_url` | TEXT | Ссылка Telegram-канал |
| `instagram_url` | TEXT | Instagram |
| `website_url` | TEXT | Сайт |
| `is_visible` | BOOLEAN | Показывать в TMA (false = скрыт) |
| `sort_order` | INTEGER | Порядок в списке спикеров |

**`sessions`** — доклады / сессии программы
> **Источник истины для всего.** Изменение любого поля сессии автоматически пересчитывает связанные рассылки.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `conference_id` | UUID → conferences | К какой конференции |
| `speaker_id` | UUID → speakers | Спикер (NULL = перерыв/техническое) |
| `track_id` | UUID → tracks | Трек — определяет ссылку на трансляцию |
| `day` | INTEGER | День конференции (1 или 2) |
| `start_datetime` | TIMESTAMP | Точная дата и время начала (с учётом timezone конференции) |
| `title` | VARCHAR | Тема доклада |
| `gift_description` | TEXT | Что спикер дарит участникам на выступлении |
| `description` | TEXT | Описание доклада (опционально) |
| `sort_order` | INTEGER | Порядок в расписании |

**`broadcast_messages`** — авто-сгенерированные рассылки
> **Ключевая таблица.** Все тексты уведомлений генерируются автоматически из данных сессии. Можно вручную отредактировать любое сообщение. При изменении сессии — не-отредактированные сообщения обновляются автоматически, отредактированные помечаются флагом "требует проверки".

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `conference_id` | UUID → conferences | К какой конференции |
| `session_id` | UUID → sessions | К какой сессии привязано (NULL = общая рассылка) |
| `type` | ENUM | `pre_5min` / `pre_30min` / `pre_1day` / `post_thanks` / `day_start` / `manual` |
| `scheduled_at` | TIMESTAMP | Когда отправить (авто = session.start_datetime - N минут, редактируемо) |
| `text` | TEXT | Текст сообщения (авто-сгенерирован, редактируемо) |
| `status` | ENUM | `draft` / `scheduled` / `sent` / `cancelled` |
| `is_edited` | BOOLEAN | true = вручную отредактировано (не перезаписывать при обновлении сессии) |
| `needs_review` | BOOLEAN | true = сессия изменилась после ручного редактирования, требует проверки |
| `sent_at` | TIMESTAMP | Фактическое время отправки |
| `recipients_count` | INTEGER | Сколько участников получили |

**`speaker_applications`** — заявки "стать спикером"
| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | Первичный ключ |
| `telegram_user_id` | UUID → telegram_users | Кто подал |
| `conference_id` | UUID → conferences | На какую конференцию |
| `name` | VARCHAR | Имя |
| `topic` | TEXT | Предлагаемая тема |
| `bio` | TEXT | О себе |
| `contact` | VARCHAR | Контакт (TG или email) |
| `status` | ENUM | `new` / `reviewed` / `accepted` / `rejected` |
| `submitted_at` | TIMESTAMP | Дата подачи |

## 10. Масштабирование на других клиентов

При развёртывании для клиента referalka (по отдельному договору):
1. Деплоится та же кодовая база с новым ботом клиента
2. В таблицу `conferences` добавляется запись для новой конференции
3. Клиент получает доступ к своей веб-админке
4. Бот — собственный бот клиента (его актив, его аудитория)
5. Схема БД идентична — переносимость без переписывания
